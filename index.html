Basis Data II

PROSES DESAIN DATABASE

1. Requirements Analysis

Langkah pertama dalam merancang aplikasi database adalah memahami data apa yang harus disimpan dalam database, aplikasi apa yang harus dibangun diatasnya, dan operasi apa yang paling sering dan cenderung pada persyaratan kinerja. Dengan kata lain, kita harus mencari tahu apa yang user inginkan dari database.

Proses identifikasi dan analisa kebutuhan-kebutuhan data disebut Requirements Analysis. Untuk menentukan kebutuhan-kebutuhan suatu sistem database, pertama-tama harus mengenal bagian-bagian lain dari sistem informasi yang akan berinteraksi dengan sistem database, termasuk user yang telah ada dan user yang baru serta aplikasi-aplikasinya. Kebutuhan-kebutuhan dari para pemakai dan aplikasi inilah yang kemudian dikumpulkan dan dianalisa

Berikut ini adalah aktifitas-aktifitas pengumpulan data dan analisa:

  1. Menentukan kelompok user dan bidang-bidang aplikasinya
  2. Peninjauan dokumentasi yang ada
  3. Analisa lingkungan operasi dan pemrosesan data
  4. Daftar pertanyaan dan wawancara

2. Conceptual Database Design

Informasi yang telah dikumpulkan dalam step Requirements Analysis digunakan untuk mengembangkan deskripsi high-level yang akan disimpan dalam database, bersamaan dengan kendala yang diketahui yang dapat menahan data ini. Langkah ini sering dilakukan dengan menggunakan model ER. Model ER adalah salah satu dari beberapa model high-level atau semantik, model data yang digunakan dalam desain database. Tujuannya yaitu untuk menciptakan deskripsi sederhana tentang data yang paling dekat dengan bagaimana user dan develop memikirkan data. Desain awal harus cukup tepat untuk memungkinkan terjemahan yang mudah menjadi model data yang didukung oleh sistem database komersial (yang dalam praktiknya, berarti model relasional).

Step Conceptual Database Design mempunyai dua aktivitas paralel, yaitu:

  1. Perancangan skema konseptual

    menguji kebutuhan-kebutuhan data dari suatu database yang merupakan hasil dari step 1, dan menghasilkan sebuah conceptual database schema pada DBMS independent model data high-level seperti EER (enhanced entity relationship) model.

  2. Perancangan transaksi

    menguji aplikasi-aplikasi database dimana kebutuhan-kebutuhannya telah dianalisa pada step 1, dan menghasilkan perincian transaksi-transaksi ini.

3. Pemilihan DBMS

Pemilihan database di tentukan oleh beberapa faktor, diantaranya : faktor teknis, ekonomi, dan organisasi.

Faktor teknis:

  1. DBMS (relational, hirarki, atau jaringan)
  2. Struktur penyimpan dan akses path yang didukung DBMS
  3. Ketersediaan interface user dan pemrogram, tipe bahasa query high-level, ketersediaan alat bantu pengembangan, kemampuan berhubungan dengan DBMS lain melalui media standard
  4. Pilihan arsitektur yang berhubungan dengan operator client-server dan lain sebagainya.

Faktor ekonomi:

  1. Software Acquisiton Cos
  2. Maintenance Cost
  3. Hardware Acquisition Cost
  4. Database Creation and Conversion Cost
  5. Personal cost
  6. Training cost
  7. Operating cost :

Faktor Organisasi:

  1. Struktur data
  2. Familiarity of Personal with the System
  3. Availability of Vendor Service

4. Logical Database Design

Kita harus memilih DBMS untuk menerapkan desain database, dan mengubah desain database serta mengubah conceptual database design menjadi skema database dalam model data dari DBMS yang dipilih. Yang dipertimbangkan hanyalah DBMS relasional, dan karena itu, tugas dalam logical design step adalah mengubah ER menjadi skema database relational. Hasilnya adalah conceptual schema, biasa disebut logical schema, dalam model data relasional.

Hasil dari step ini memakai perintah-perintah DDL dalam bahasa DBMS yang dipilih yang menentukan tingkat skema konseptual dan eksternal dari sistem database. Tetapi dalam beberapa hal, perintah-perintah DDL memasukkan parameter-parameter rancangan fisik sehingga DDL yang lengkap harus menunggu sampai Physical Database Design step telah lengkap.

Proses pemetaan ada dua bentuk :

  1. Pemetaan yang Tidak Tergantung pada Sistem (SystemIndependet Mapping )

    Pada bentuk ini, pemetaan tidak mempertimbangkan karakteristik khusus atau kasus khusus yang diaplikasikan ke implementasi DBMS dari model data.

  2. Penyesuaian Skema ke DBMS yang Spesifik (Tailoring the Schemas to Specific DBMS)

    DBMS yang berbeda mengimplementasikan model data dengan menggunakan pemodelan khusus. Hasilnya merupakan pernyataan DDL dari DBMS yang dipilih

5. Physical Database Design

Proses pemilihan struktur penyimpanan dan jalur akses pada file-file database untuk mencapai penampilan yang terbaik pada bermacam aplikasi. Dirancang secara spesifikasi untuk database yang disimpan yang berhubungan dengan struktur-struktur penyimpanan fisik, penempatan record dan jalur akses.

Beberapa petunjuk dalam pemilihan physical database design:

  1. Waktu Respon

    Waktu transaksi database untuk menerima respon selama eksekusi. Waktu respon dipengaruhi oleh waktu akses database untuk data item yang ditunjuk oleh suatu transaksi.

  2. Penggunaan Ruang Penyimpanan

    Jumlah ruang penyimpanan yang digunakan oleh file database dan struktur- struktur jalur akses.

  3. Transaction Throughput

    Rata-rata jumlah transaksi yang dapat diproses per menit oleh sistem database, dan merupakan parameter kritis dari sistem transaksi.

6. Application and Security Design

Setiap project perangkat lunak yang melibatkan DBMS harus mempertimbangkan aspek aplikasi yang melampaui database itu sendiri. Setiap peran dalam database, harus diidentifikasi bagian-bagian mana yang boleh diakses dan tidak boleh diakses dan harus dipastikan bahwa peraturan akses ini berlaku.
